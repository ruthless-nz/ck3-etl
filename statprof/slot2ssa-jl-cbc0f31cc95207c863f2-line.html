239 # Run iterated dominance frontier
240 #
241 # The algorithm we have here essentially follows LLVM, which itself is a
242 # a cleaned up version of the linear-time algorithm described in
243 #
244 # A Linear Time Algorithm for Placing phi-Nodes (by Sreedhar and Gao)
245 #
246 # The algorithm here, is quite straightforward. Suppose we have a CFG:
247 #
248 # A -> B -> D -> F
249 # \-> C -------/
250 #
251 # and a corresponding dominator tree:
252 #
253 # A
254 # |- B - D
255 # |- C
256 # |- F
257 #
258 # Now, for every definition of our slot, we simply walk down the dominator
259 # tree and look for any edges that leave the sub-domtree rooted by our definition.
260 #
261 # E.g. in our example above, if we have a definition in `B`, we look at its successors,
262 # which is only `D`, which is dominated by `B` and hence doesn't need a phi node.
263 # Then we descend down the subtree rooted at `B` and end up in `D`. `D` has a successor
264 # `F`, which is not part of the current subtree, (i.e. not dominated by `B`), so it
265 # needs a phi node.
266 #
267 # Now, the key insight of that algorithm is that we have two defs, in blocks `A` and `B`,
268 # and `A` dominates `B`, then we do not need to recurse into `B`, because the set of
269 # potential backedges from a subtree rooted at `B` (to outside the subtree) is a strict
270 # subset of those backedges from a subtree rooted at `A` (out outside the subtree rooted
271 # at `A`). Note however that this does not work the other way. Thus, the algorithm
272 # needs to make sure that we always visit `B` before `A`.
